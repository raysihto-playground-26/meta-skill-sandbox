# plan-r1 集約分析レポート（日本語）

## 1. 目的と対象

本レポートは、以下 5 本の計画書を比較し、**各プランを特徴づける他にない特色**を重視して集約したものである。

- `docs/.tmp/plan-r1.en.claude.md`
- `docs/.tmp/plan-r1.en.codex.md`
- `docs/.tmp/plan-r1.en.copilot.md`
- `docs/.tmp/plan-r1.en.cursor.md`
- `docs/.tmp/plan-r1.en.gpt.md`

前提として、課題定義は `docs/.tmp/report_p1_p2_gap_analysis.md`（P1: conflict 解決矛盾、P2: condition 識別子語彙不整合）に依拠する。

## 2. 5 プランの全体像（要約）

| プラン | 基本スタンス | 具体性 | 検証設計 | 主な固有性 |
| --- | --- | --- | --- | --- |
| claude | 仕様文言を直接改訂する前提 | 非常に高い（提案文言・変更ID・影響棚卸しあり） | 受入基準・段階実行・リスク表が詳細 | **全変更を「差分候補」としてほぼ実装可能な粒度まで明示** |
| codex | 段階的に整合性を回復する前提 | 中程度（方針中心） | フェーズ/Exit条件/AC が明確 | **Phase 0 の在庫化（識別子マッピング）を起点にする運用設計** |
| copilot | 直接改訂前提 + 代替案比較 | 高い（推奨案と代替案を併記） | フェーズ受入条件とリスクが具体 | **P2 で「推奨案/代替案」の二経路を同時提示** |
| cursor | 直接改訂前提の簡潔プラン | 中程度（必要更新点を簡潔列挙） | 実装順序・成功条件を簡潔に提示 | **最短で意思決定できるコンパクトな仕様更新チェックリスト型** |
| gpt | 今回は AI directive files を編集しない前提 | 高い（ただし対象は runner/linter 側） | アルゴリズム・テスト・レポート出力まで具体 | **ファイル修正禁止制約下で runner/linter 側に解決責務を移す設計** |

## 3. プラン別分析（他にない特色を中心に）

### 3.1 `plan-r1.en.claude.md`

**特色（固有性）**
- P1/P2 を `P1-A`〜`P2-D` の変更単位へ分解し、**現行文言と提案文言を対で提示**している。
- P2 について、影響対象ルールを大きな一覧で明示し、`conditions` 正規化の適用範囲をほぼ網羅的に示している。
- 実行計画に tier 分離を導入し、`objective validation` と `subjective review` を層で分ける設計を明記している。

**強み**
- そのまま編集作業に着手できるほど具体度が高い。
- 受入基準（P1-AC*, P2-AC*）とリスクが対応づけられており、実行後判定がしやすい。

**留意点**
- 変更提案が広範囲（conditions 正規化）であるため、実施時は機械的検証の運用が前提になる。

---

### 3.2 `plan-r1.en.codex.md`

**特色（固有性）**
- `Phase 0` で「conditions 識別子」と「definitions キー」の**1対1マッピング在庫**を必須化している。
- 各フェーズに Exit condition を設定し、進行管理を工程ベースで行う構成になっている。

**強み**
- 実作業前に語彙不整合を棚卸しするため、P2 の取りこぼし抑制に向く。
- 文言修正の詳細に踏み込みすぎず、実装手段をチーム側で選びやすい。

**留意点**
- 具体的な置換文言・キー命名案は他案より少なく、実編集時に追加設計が必要になる。

---

### 3.3 `plan-r1.en.copilot.md`

**特色（固有性）**
- P2 で **Approach 1（推奨）** と **Approach 2（代替）** を並列提示し、選択理由（保守性・可読性）まで記述している。
- P1-B（`interpretation.priority` の明確化）を optional として明示し、最小修正と明確化修正を分離して判断できる構造。

**強み**
- 複数の実装経路を比較したうえで推奨案へ誘導するため、合意形成に使いやすい。
- 影響規模（~30 ルールなど）を示し、作業コスト感を事前把握しやすい。

**留意点**
- 代替案を許容する設計のため、採用方針を先に固定しないと実装が分岐しやすい。

---

### 3.4 `plan-r1.en.cursor.md`

**特色（固有性）**
- 5 セクション構成で、問題→更新点→順序→成功条件が短くまとまった**コンパクト設計**。
- P2 でスコープ識別子の揃え方に `(a)/(b)` の選択肢を示し、最小意思決定で前進できる。

**強み**
- 速く読む・速く着手する用途に強い。
- 成功条件が明快で、レビューの観点がぶれにくい。

**留意点**
- 詳細な検証手順や影響棚卸しは薄く、大規模置換時には補助ドキュメントが必要になる。

---

### 3.5 `plan-r1.en.gpt.md`

**特色（固有性）**
- 冒頭で「この反復では AI directive files を編集しない」と制約を固定し、**runner/linter 側で決定論を担保**する計画へ転換している。
- P2 では、`scope ...` の空白/ハイフン差を alias 解決しつつ、未定義識別子を明示報告する運用を提案している。
- テスト計画と機械可読レポート出力（未解決/alias 使用の分類）まで含め、運用面に強い。

**強み**
- ファイル修正ができない条件下でも、判定の一貫性を先に確立できる。
- 実行系（runner/linter）の挙動を先に固定でき、再発監視に向く。

**留意点**
- P1/P2 の根本（仕様本文の自己矛盾・自己違反）そのものは、ファイル未編集のまま残る。

## 5. まとめ（集約所見）

1. **仕様本文を直接整える計画として最も完成度が高いのは `claude`**  
   文言差分、受入基準、影響棚卸し、リスクが一体化している。

2. **運用しやすい工程設計としては `codex` が明快**  
   Phase 0 在庫化を先頭に置くため、P2 の取りこぼし防止に有効。

3. **意思決定支援（複数案比較）では `copilot` が強い**  
   推奨案/代替案の比較により、採用理由を明文化しやすい。

4. **最短で合意形成したい場面では `cursor` が有効**  
   簡潔なため、初回レビューやキックオフ向け。

5. **編集禁止制約下での実効性では `gpt` が唯一の選択肢**  
   仕様本文改訂ではなく、runner/linter 側で決定論と可観測性を先行実装する発想が他案にない。

総じて、5案は「**仕様本文を直す路線（claude/codex/copilot/cursor）**」と「**実行系で先に吸収する路線（gpt）**」に分かれる。  
特に固有性の観点では、`claude` の高密度差分設計と、`gpt` の runner/linter 中心設計が両極として際立っている。
