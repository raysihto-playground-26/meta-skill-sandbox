## 0. 前提（ギャップ分析レポートの要点）

本レポートが対象とする課題は、ギャップ分析レポートが提示する以下の2点である。

- **P1（Critical）**: `interpretation.priority`（同一layer内は数値priorityの高い方が勝つ）と、`precedence_and_conflict.conflict_policy.MUST_vs_MUST`（MUST同士は停止/確認要求）が、**同一layerでpriorityが異なるMUST同士の衝突**において両立せず、runnerの挙動が不定になる。
- **P2（High）**: `interpretation.compound_conditions` が「条件識別子はファイル内またはdefinitionsに定義されるMUST」を要求している一方で、`conditions` で使われる識別子が **未定義**、または **スペース区切り vs ハイフン区切り**の不一致により、機械的照合が破綻している（自己違反）。

ギャップ分析レポートが示す To-Be は、概ね次の形に整理される。

- **P1（決定論）**: layer → priority →（同一layer・同一priorityのタイのみ）conflict policy の順で解決できるよう、適用条件を分離する。
- **P2（語彙の整備）**: `conditions` で使う識別子を定義し、`conditions` と `definitions` が **同一の正規形**で一致して機械検証できるようにする。

## 1. 5本の plan-r1 の共通点（収束している「解決の骨格」）

5本すべてが、P1/P2 を「**runnerが決定論的に評価できる仕様**」という観点で捉えている点は共通している。

- **P1**: layer優先、同一layerならpriority優先、どちらでも決着しないタイのみ `MUST_vs_MUST` の停止/確認要求、という **3段の決定手順**を明示する方向に収束している。
- **P2**: `conditions` のトークンを `definitions` に照合可能にする（未定義の解消、トークン形式の統一、compound（`"A and B"`）の分解・検証）という **機械的検証可能性**を中核に据えている。

一方で、**「どこを変更対象とみなすか」**と **「どこまで具体化するか」**が、各プランの差分（特色）になっている。

## 2. 各プランの「他にない特色」（差分の核）

### 2.1 `plan-r1.en.claude.md` の特色

- **仕様テキストの“差し替え案”が最も具体的**: P1で `MUST_vs_MUST` と `interpretation.priority` の文字列を書き換える提案を、現行/提案テキスト（YAML断片）として明示している。
- **「ルールレコード（id/layer/priority/…）」という内部スキーマを固定条件として扱う**: 変更制約として「既存のrule-record schemaを保持」「YAMLパース成功」を強く前提化している。
- **P2を“正規形（hyphen-lowercase）”へ全面正規化する設計**: `definitions` へ不足識別子を追加し、`conditions` 側も同じ正規形へ寄せる方針を一貫して書く。さらに `interpretation` に **識別子形式の規約キー**（例: `condition_identifiers`）を追加する提案で、再発防止まで踏み込む。
- **影響範囲の棚卸しが唯一「網羅リスト」形式**: どのルールがどの `conditions` を使っており、どこが書き換え対象かを「影響を受けるルール一覧」として列挙している（作業漏れリスクを直接減らす構成）。
- **実行計画が“tier-separation”原則に沿って細分化**: フェーズ＋tier（Tier 0〜）で「出力形式」「停止条件」まで定義し、検証手順も具体化している。

### 2.2 `plan-r1.en.codex.md` の特色

- **最小構成で“工程の骨組み”を先に固める**: P1/P2の要点を整理したうえで、Phase 0（識別子インベントリ）→ Phase 1（P1）→ Phase 2（P2）→ Phase 3（検証整備）という、**出口条件付きの工程**に圧縮している。
- **「Phase 0: inventory/mapping」を中核に据える**: 使用識別子・定義識別子・一致/別名/欠落の状態を 1:1 マッピング表にすることを、最初の成果物として強調している。
- **具体的な文言案や命名規約の提示は控えめ**: 代わりに「決定論と検証可能性」という狙いを工程に落とすことに集中している。

### 2.3 `plan-r1.en.copilot.md` の特色

- **P2を“二つのアプローチ比較”として提示する**: 推奨（Approach 1: `conditions` を `definitions` の形式へ正規化）と代替（Approach 2: `definitions` をスペース形式へ寄せる）を明示し、代替案の弱点（スペースを含むYAMLキーの扱い等）も含めて判断材料を提供する。
- **「変更規模（Change size）」の見積り観点が明確**: P1は文字列値の書換え、P2は多数の `conditions` 変更がある、というリスクの源泉を表に落としている。
- **将来検討（future consideration）を具体項目で列挙**: 例として `triggers` セクション導入、compound条件の形式文法（BNF等）の導入など、P2の語彙拡張を見越した整理案が含まれる。

### 2.4 `plan-r1.en.cursor.md` の特色

- **“最短の実装順序＋成功条件”に強くフォーカス**: P1→P2→両者の検証、という手順と成功条件を短い文章・表で提示し、重い前提定義や詳細なリスク評価を省いている。
- **P2の正規形選択を「分岐として残す」**: scope識別子の統一を「(a) conditionsをハイフンへ」「(b) スペース形式を定義してそちらへ」など、選択肢提示に留め、推奨を強く固定しない。

### 2.5 `plan-r1.en.gpt.md` の特色

- **“AI directive files を編集しない”制約を前提に、解決対象を runner/linter 側へ移す**: 他プランが主に SKILL.md のYAMLを整える方向なのに対し、本プランは「現状が不整合でも評価を決定論にする」ために、**実装（runner/linterの挙動）を成果物にする**点が際立つ。
- **P1はアルゴリズム仕様＋テストを成果物化**: layer→priority→（タイのみ）policy の手順を「実装志向の仕様」として書き、さらにその分岐を通す **テスト/フィクスチャ**を計画に含める。
- **P2は“解決（resolution）手順”を定義し、既知の不一致を決定論的に扱う**: scopeのスペース/ハイフン不一致に対し、空白→ハイフンの別名解決を「決定論的aliasルール」として設け、どの識別子が alias で解決されたかを報告する。
- **未定義識別子を「予約トリガーの明示リスト」等で扱う案を提示**: 任意の未定義トークンを黙認せず、ギャップ分析レポートで列挙された識別子に限定して“制御された扱い”にする、という制約付き方針を含む。
- **機械可読なレポート出力を成果物に含める**: 未解決・alias解決・集計を出すことで、P2の“照合不能”を運用上の観測可能性に転換する設計になっている。

## 4. 統合所見（5案を束ねると何が見えるか）

- **「仕様テキストを直して自己整合にする」系（claude / copilot / cursor / codex）**は、P1/P2 を SKILL.md 内の自己矛盾として捉え、正規形の統一と受け入れ基準により **仕様そのものを決定論へ戻す**ことを狙う。
  - 特に `plan-r1.en.claude.md` は、変更内容・影響範囲・検証を最も“編集手順に落ちる形”で提供する。
  - `plan-r1.en.copilot.md` は、P2で「どちらに寄せるか」の意思決定を前面に出し、将来整理（`triggers` 等）も含む。
  - `plan-r1.en.codex.md` は、最小限の工程設計により、作業の進め方と出口条件を先に固定する。
  - `plan-r1.en.cursor.md` は、短い順序と成功条件で全体像を掴ませる用途に寄る。
- **「編集禁止を前提に実装側で吸収する」系（gpt）**は、P1/P2 を “現状の不整合をどう安全に評価するか” に再定義し、アルゴリズム・テスト・機械可読レポートで **運用上の決定論**を担保しようとする点が独自である。
