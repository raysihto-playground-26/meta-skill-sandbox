## 0. 本レポートの目的と前提

### 0.1 目的

本レポートは、`docs/.tmp/report_p1_p2_gap_analysis.md` で特定された以下の 2 つの課題:

| ID  | 課題                                                                  | 重要度   |
| --- | --------------------------------------------------------------------- | -------- |
| P1  | `interpretation.priority` と `MUST_vs_MUST` の矛盾 (衝突解決の不定)  | Critical |
| P2  | condition 識別子の語彙不在 + トークン形式の不一致 (MUST 自己違反)     | High     |

に対する是正プラン 5 本を比較分析し、各プランの特徴、とりわけ他のプランにはない固有の特色を明らかにするものである。

### 0.2 前提

- 本レポートの内容は、上記 5 本のプランファイルおよびギャップ分析レポートに記載された情報のみに基づく。
- AI directive files の具体的な変更は本レポートの対象外である。
- 各プランファイルの名前は、以降それぞれ **Claude**, **Codex**, **Copilot**, **Cursor**, **GPT** と略記する。

### 0.3 課題の概要 (ギャップ分析レポートより)

**P1: 衝突解決の曖昧性**

`interpretation.priority` は「同一 layer 内では数値 priority が高い方が勝つ」と記述している一方、`precedence_and_conflict.conflict_policy.MUST_vs_MUST` は「Halt or request clarification」を無条件に要求している。同一 layer で priority が異なる 2 つの MUST ルールが衝突した場合、runner はどちらの指示に従うべきか判断できない。

**P2: condition 識別子語彙の問題**

`interpretation.compound_conditions` は条件識別子の定義を MUST で要求しているが、実際に `conditions` 配列で使用されている多数の識別子 (`creating AI directive file`, `scope high-stakes` 等) が `definitions` に定義されていない。さらに、スコープ識別子のトークン形式が `conditions` ではスペース区切り、`definitions` ではハイフン区切りとなっており、機械的照合が失敗する。

## 2. 各プランの固有の特色

### 2.1 Claude — 最も詳細・網羅的な設計書

**固有の特色:**

1. **Change P2-D: `interpretation` への `condition_identifiers` 規約の追加提案**

   Claude のみが、識別子のトークン形式に関する規約を `interpretation` セクションの新しいキーとして明示的に追加することを提案している。具体的な提案内容は:

   ```yaml
   condition_identifiers: "Condition identifiers in conditions arrays MUST use
     hyphen-separated lowercase form matching the corresponding definitions key.
     When a compound condition uses the 'A and B' form, both A and B MUST be
     hyphen-separated identifiers defined in definitions."
   ```

   他のプランは既存セクションの修正に留まるが、Claude は将来の不整合を防止するための規約を仕様自体に埋め込むことを提案している点で独自性がある。

2. **影響ルール全量リスト (Section 2.4)**

   Claude は P2 の影響を受ける全ルール (約 44 ルール) を 3 カテゴリに分けて完全列挙している。これにより、変更の漏れを防ぎ、実装時のチェックリストとして直接利用できる。他のプランでは影響範囲をカテゴリ単位で記述するに留まっている。

3. **`failure_states_and_degradation` との整合性の確認 (P1)**

   P1 の設計根拠 (Section 1.3) において、`failure_states_and_degradation.failure_states` の `rule-conflict` エントリが既に「Apply precedence_and_conflict; if unresolved MUST halt or request clarification」と記述している点を確認し、提案する変更がこの既存記述と整合することを明示している。他のプランではこのクロスリファレンスは行われていない。

4. **P2 のティア分離を 6 段階に細分化**

   Phase 2 の実行計画において、Tier 0 (棚卸し) から Tier 5 (主観的レビュー) まで 6 段階に細分化している。特に Tier 1 (定義追加), Tier 2 (条件正規化), Tier 3 (規約追加) を分離している点は他のプランにない粒度である。

5. **`plan.final.ascii.md` との相互作用リスクの評価**

   リスク評価 (Section 4) において、本プランの変更が `plan.final.ascii.md` で計画されている MUST NOT セクション是正と構造的に衝突しないことを明示的に確認している。他のプランではこの既存プランとの相互作用は言及されていない。

### 2.2 Codex — 抽象度の高い段階的フレームワーク

**固有の特色:**

1. **Phase 0: ベースラインインベントリの独立フェーズ化**

   Codex は他のプランが前提確認を Tier 0 として扱うのに対し、全 condition 識別子の棚卸し (inventory) を独立した Phase 0 として位置づけている。具体的には、使用済み識別子と定義済み識別子の 1 対 1 マッピングテーブルを作成し、各識別子の状態を「exact match / alias only / missing」の 3 分類で管理することを提案している。

   この「マッピングテーブル + 3 分類」のアプローチは他のプランにはなく、棚卸し結果が Phase 1・2 の前提条件 (exit condition) となる点で、フェーズ間の依存関係が最も明確に設計されている。

2. **Phase 3: 検証アラインメントの独立フェーズ化**

   Codex のみが、是正後の検証メカニズム自体の更新・追加を独立した Phase 3 として定義している。他のプランでは検証はティアの一部として組み込まれているが、Codex は「P1/P2 のリグレッションを検出できる検証ロジックをファイル自体に含めること」を明示的な exit condition として要求している。

3. **具体的 YAML 不提示の設計方針**

   Codex は具体的な YAML の変更案を意図的に提示せず、各 Phase の「exit condition」のみで要件を定義している。これにより、実装者の判断余地を最大化する抽象的なフレームワークとなっている。

### 2.3 Copilot — 代替案を含む判断根拠重視の構成

**固有の特色:**

1. **P2 に対する 2 つのアプローチの提示と比較**

   Copilot は P2 の是正方針として 2 つのアプローチを明示的に比較している:

   - **Approach 1 (推奨)**: `conditions` 側をハイフン区切りに正規化し、`definitions` のスタイルに合わせる。
   - **Approach 2 (代替)**: `definitions` 側をスペース区切りに変更し、`conditions` に合わせる。

   Approach 2 に対しては「YAML キーにスペースを含めるとクォーティングが必要で保守性が低下する」という具体的なデメリットを述べた上で Approach 1 を推奨している。他のプランはいずれもハイフン正規化の方向性を採用しているが、代替案を明示的に列挙し却下理由を述べているのは Copilot のみである。

2. **P1-B を「optional」として明示**

   `interpretation.priority` への前方参照追加 (P1-B) を「optional, for clarity」と明記しており、必須変更と任意改善の区別が明確である。Claude も同様の変更を提案しているが「optional」とは明記していない。

3. **リスクの「現時点での安全性」に関する具体的論証**

   リスクテーブルにおいて、「conditions 文字列の変更が runner の動作に影響する可能性」に対して「現在の識別子は未定義であるため、準拠している runner はこれらに依拠できない。従って正規化は安全である」という論理的帰結を示している。この「未定義であること自体が安全性の根拠になる」という議論は他のプランにはない。

### 2.4 Cursor — 最も簡潔で実装指向

**固有の特色:**

1. **最小限の構成要素による直線的なステップ構成**

   Cursor は目的・スコープ・非目標といったメタセクションを大幅に省略し、Issue Summary → Resolution Strategy → Required Updates → Implementation Order → Success Criteria の 5 セクション構成としている。これにより、全体が 154 行で完結し、5 本中最も簡潔なプランとなっている。

2. **「Required Specification Updates」の直接列挙**

   他のプランが「Change P1-A」「Change P2-B」のようにラベル付きの変更案を定義するのに対し、Cursor は必要な仕様更新をリスト形式で直接列挙している。例えば P2 について:

   - 4 つのトリガー識別子の定義追加
   - スコープ識別子のアライン方法 (a)(b) 2 択の提示
   - 全識別子の文字列完全一致の確保

   この箇条書きスタイルは、レビュー者が「何を変えるか」を素早く把握するのに適している。

3. **P2 の正規化方向について 2 択を提示**

   Copilot と同様に P2 のトークン正規化方向について (a) ハイフン形式に統一 と (b) スペース形式に統一 の 2 つの選択肢を提示しているが、Copilot とは異なり推奨を明示していない。最終的な判断を実装者に委ねる構成となっている。

### 2.5 GPT — runner/linter 側の振る舞い設計に特化

**固有の特色:**

1. **「AI directive files を編集しない」制約下での runner/linter 側の解決策**

   GPT のみが、「AI directive files の具体的変更は本イテレーションでは禁止」という制約を、仕様側ではなく **runner/linter 側の振る舞い**で課題を解決するアプローチとして解釈している。他の 4 プランは全て SKILL.md 自体の YAML を修正する方向で計画しているのに対し、GPT は runner/linter のアルゴリズムや検証ロジックとしての実装を計画している。

   これは本質的に異なるアプローチであり、5 本の中で最も独自性が高い特色である。

2. **エイリアス解決メカニズムの提案 (P2)**

   P2 のトークン形式不一致に対し、他のプランが「条件側を正規化する」アプローチを採るのに対し、GPT は runner/linter 側に**エイリアス解決ルール**を実装することを提案している:

   - まず定義キーとの完全一致を試みる
   - 失敗した場合、既知のスペース→ハイフン変換ルールを適用して再照合する
   - それでも失敗した場合、未定義として報告する

   このアプローチでは仕様ファイル自体は変更せず、不整合を runner 側で吸収する。

3. **P1 に対するテストケースの作成を明示的に計画 (Section 2)**

   GPT は P1 の衝突解決アルゴリズムに対して、以下の 3 つのテストケースの作成を明示的に要求している:

   - 異なる layer でのルール衝突 (layer 優先で決定)
   - 同一 layer・異なる priority (priority で決定)
   - 同一 layer・同一 priority での MUST vs MUST (halt パス)

   テストケースの作成を是正プランの成果物として要求しているのは GPT のみである。

4. **「予約トリガー」の概念導入**

   P2 の未定義識別子に対し、GPT は runner/linter がこれらを「controlled set of known, reserved triggers」として扱うことを提案している。これにより、定義が存在しない識別子を一律にエラーとするのではなく、既知の予約識別子として明示的に許容リストを管理するアプローチを取る。

## 3. 比較分析: アプローチの分類

5 本のプランは、大きく 2 つのアプローチに分類できる:

### 分類 A: 仕様ファイル修正アプローチ (Claude, Codex, Copilot, Cursor)

SKILL.md 自体の YAML 内容を修正して整合性を回復する方向。4 本全てが以下を共有:

- `MUST_vs_MUST` の適用条件を同一 layer・同一 priority の場合に限定する
- 未定義の condition 識別子を `definitions` に追加する
- トークン形式をハイフン区切りに統一する (Cursor は選択肢を提示)

### 分類 B: runner/linter 側解決アプローチ (GPT)

SKILL.md を変更せず、runner/linter の振る舞いで不整合を吸収する方向。以下を特徴とする:

- 衝突解決アルゴリズムを runner 側に明文化
- エイリアス解決で トークン形式の不一致を吸収
- 未定義識別子を予約トリガーとして管理

### アプローチ比較

| 観点                       | 分類 A (仕様修正)                              | 分類 B (runner/linter 解決)                        |
| -------------------------- | ---------------------------------------------- | -------------------------------------------------- |
| 根本解決                   | 仕様の自己違反を直接解消                       | 仕様は不変; runner が矛盾を吸収                    |
| 機械的検証の単純性         | 定義と条件の完全一致で検証可能                 | エイリアス規則の正当性を別途保証する必要あり       |
| 将来の保守性               | 仕様自体が整合的であり保守が容易               | runner のエイリアス規則を仕様と同期させる負担あり  |
| 実装負荷                   | 多数のルール (~30〜44) の `conditions` を書換え | runner/linter のコード実装が必要                    |
| 仕様変更禁止制約との整合性 | 「プラン段階」では準拠; 実行時に変更が必要     | プラン段階・実行段階ともに仕様ファイル不変で完結   |

### 5.2 補完関係

5 本のプランは互いに排他的ではなく、以下のような補完関係がある:

- **Claude の全量リスト** は、他の 4 プランいずれを採用する場合でも実装時のチェックリストとして活用できる。
- **Codex の Phase 0 (棚卸し)** は、どのアプローチを採用するにせよ最初に実施すべき手順として共通に有用である。
- **Copilot の代替案比較** は、実装方針の意思決定における判断材料として機能する。
- **GPT のエイリアス解決メカニズム** は、仕様修正が段階的に行われる場合の過渡的な緩和策として分類 A のプランと併用可能である。
- **GPT のテストケース計画** は、分類 A のプランで仕様修正を行った後の検証にも直接活用できる。
- **Cursor の簡潔な構成** は、レビュー者が短時間で方針を把握する際のサマリーとして有用である。
