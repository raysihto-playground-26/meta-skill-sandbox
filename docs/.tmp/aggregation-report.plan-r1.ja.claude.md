## 2. 全体概観

### 2.1 アプローチの分類

5 本のプランは、大きく 2 つのアプローチに分かれる。

**アプローチ A: SKILL.md 自体の修正を計画するもの（4 本）**

Claude、Codex、Copilot、Cursor の 4 プランは、いずれも SKILL.md の YAML 内容を直接修正することで P1/P2 を解決する方針を採っている。ただし、具体性のレベルと設計判断には大きな差がある。

**アプローチ B: ランナー／リンター側の挙動で吸収するもの（1 本）**

GPT プランのみが、「AI directive files の具体的変更は今回のイテレーションでは行わない」という制約を自ら設定し、ランナー／リンター側のロジックで問題を吸収する方針を採っている。これは他の 4 本とは根本的に異なる設計哲学であり、本レポートにおいて最も注目すべき差異である。

## 3. 各プランの特徴分析

### 3.1 Claude プラン

**全体的性格**: 最も網羅的・詳細で、実装者が追加判断をほぼ必要としない「実装仕様書」型のプラン。

**他にない特色**:

1. **影響ルール完全棚卸し（Section 2.4）**: P2 の conditions 変更が及ぶ全ルール（約 44 件）を、3 カテゴリに分類して個別に列挙している。他のプランはいずれも「約 30 ルール」「ほとんどのルール」等の概数表現に留まるが、Claude プランだけが全ルール ID を明示している。実装時の漏れ防止に直接寄与する。

2. **Change P2-D: `interpretation` への正規形規約の追加提案**: 識別子のトークン形式を `interpretation.condition_identifiers` として明文化し、将来の不整合を予防する仕組みを提案している。これは問題の「修正」にとどまらず「再発防止」を志向する設計であり、他のどのプランにもない。Copilot プランが「将来の検討事項」として formal grammar を挙げているのと対照的に、Claude プランは具体的な YAML テキストまで提示している。

3. **衝突解決カスケード図（Section 1.4）**: P1 修正後の 3 段階解決手順を、疑似コード風のフローチャートとして独立セクションに記載している。他プランも同等のロジックを記述しているが、独立したアルゴリズム表現として切り出しているのは Claude プランのみである。

4. **リスク評価の定量性（Section 4）**: Likelihood / Impact の 2 軸マトリクスで 4 件のリスクを評価している。他プランのリスク記述はリスト形式のみである。

**制約・注意点**: 分量が大きいため、レビュー負荷が高い。また、P2-D の `condition_identifiers` 追加は新たなキーを `interpretation` に導入するため、既存の `interpretation` スキーマとの整合性検討が必要になる。

### 3.2 Codex プラン

**全体的性格**: 最も抽象度が高く、プロセス重視の「方針書」型プラン。

**他にない特色**:

1. **Phase 0「ベースライン棚卸し」の明示**: 変更実行前に、すべての条件識別子について「使用中の識別子 / 定義済み識別子 / ステータス（完全一致 / エイリアスのみ / 未定義）」のマッピングテーブルを作成するフェーズを独立に設けている。他のプランは棚卸しを暗黙的に前提とするか（Claude は結果を 2.4 に記載）、言及しないが、Codex プランだけが「棚卸し自体をフェーズとして定義し、その完了を exit condition とする」構造を持つ。これは品質管理プロセスとして形式的に正しい。

2. **Phase 3「検証整合性」の独立フェーズ化**: 変更後に SKILL.md 内の `verification` セクションを更新し、P1/P2 の回帰を検出可能にすることを独立フェーズとして要求している。他プランでは検証は受入基準の一部として扱われるが、Codex は検証機構の更新自体を成果物として位置付けている。

3. **具体的 YAML テキストを一切含まない**: 意図的に抽象レベルにとどまり、「何をすべきか」を定義して「どう書くか」は実装者に委ねる方針。これにより、プラン自体の寿命が長くなる（SKILL.md の微細な文言変更に影響されない）。

**制約・注意点**: 具体性が低いため、実装者は別途詳細設計を行う必要がある。受入基準も 6 項目に集約されており、検証の具体的手順は定められていない。

### 3.3 Copilot プラン

**全体的性格**: 代替案を提示し、推奨を明示する「意思決定支援」型プラン。

**他にない特色**:

1. **P2 に対する 2 つのアプローチの提示（Section 2.3）**: トークン形式の統一方法として、Approach 1（conditions 側をハイフン区切りに正規化）と Approach 2（definitions 側をスペース区切りに正規化）の 2 案を提示し、Approach 2 の欠点（YAML キーのクォーティング問題）を根拠に Approach 1 を推奨している。他のプランはすべてハイフン区切りへの正規化を前提としており、代替案を明示的に検討・棄却するプロセスを記載しているのは Copilot プランのみである。

2. **大文字保持のハイフン区切り識別子**: P2 の識別子正規化において、`creating-AI-directive-file`、`creating-AI-directive-file-that-contains-YAML-block` のように、「AI」「YAML」の大文字を保持するハイフン区切り形式を採用している。Claude プランの `creating-ai-directive-file`（全小文字）とは異なる設計判断であり、固有名詞の可読性を重視した選択と解釈できる。ただし、既存の definitions キー（`scope-high-stakes` 等）は全小文字であるため、ファイル内での一貫性には課題が残る。

3. **将来検討事項の広さ（Section 5）**: 4 項目の将来検討事項を挙げており、特に以下は他プランにない視点である:
   - `MUST_vs_SHOULD` と `prohibition_vs_other` への P1 修正の波及検討（P1 の修正範囲を他の conflict policy エントリにも拡張すべきかという問い）
   - 複合条件の BNF 形式文法の定義（条件文字列の完全な機械的パース可能性の追求）

4. **Change P1-B を「optional」と明記**: `interpretation.priority` への前方参照追加を任意としている点は、他プランが必須としているのと対照的である。

**制約・注意点**: 大文字保持の識別子命名は、既存のハイフン区切りキー（全小文字）との混在を生む可能性がある。

### 3.4 Cursor プラン

**全体的性格**: 最も簡潔・実用的な「チェックリスト」型プラン。

**他にない特色**:

1. **最小限の構成による高い可読性**: 約 155 行で P1/P2 の問題、解決戦略、必要な仕様更新、実装順序、成功基準をすべてカバーしている。冗長性を排除し、要点のみを列挙するスタイルは、短時間でのレビューと合意形成に適している。

2. **P2 のスコープ識別子について 2 案を並記するが推奨を示さない（Section 3.3）**: (a) ハイフン形式を conditions に合わせる案と (b) スペース形式を definitions に合わせる案を列挙しつつ、どちらを推奨するかの判断を保留している。Copilot プランが明確に推奨を示すのと対照的で、意思決定を読者に委ねる設計である。

3. **ティア分離やフェーズ分割を行わない**: 他の 4 プランがいずれもフェーズ分割やティア分離実行を採用しているのに対し、Cursor プランは単純な 5 ステップの実装順序テーブルのみを提示している。これは実装のオーバーヘッドを最小化する設計と解釈できる。

**制約・注意点**: 具体的な YAML テキストを含まないため、実装者による解釈の余地が大きい。リスク評価が存在しない。

### 3.5 GPT プラン

**全体的性格**: 唯一のランナー／リンター側解決を志向する「インフラ設計」型プラン。

**他にない特色**:

1. **SKILL.md の直接編集を行わないという根本方針**: 「Concrete edits to AI directive files are not allowed in this iteration」という制約を自ら設定し、ランナー／リンター側のロジックで P1/P2 を吸収する方針を採っている。これは他の 4 本とは根本的に異なる。この方針には以下の含意がある:
   - SKILL.md の内部矛盾は残存するが、ランナーの決定論的挙動が矛盾を上書きする
   - ランナー仕様がドキュメント化されることで、矛盾が存在する状態でも運用可能になる

2. **エイリアシング機構の提案（Section 3）**: P2 のスペース／ハイフン不一致に対し、識別子の正規化ではなく、ランナー側でのエイリアス解決規則を提案している。具体的には「T が "scope " で始まりスペースを含む場合、スペースをハイフンに置換して再照合する」という決定論的なエイリアス規則である。他のプランがすべて「識別子の正規化（SKILL.md 側の修正）」で問題を解決するのに対し、GPT プランはランタイム側の互換層で吸収する。このアプローチのリスクとして、プラン自身が「エイリアシングが本来の語彙のずれを隠蔽する可能性」を挙げている。

3. **未定義識別子の「予約トリガー」概念**: SKILL.md の definitions に存在しない条件識別子（`creating AI directive file` 等）を、ランナー側の「予約トリガーの明示的リスト」として管理する方針を提案している。これは SKILL.md の自己違反を「修正」するのではなく「容認した上で制御下に置く」アプローチである。

4. **テスト駆動（Section 2）**: P1 の衝突解決アルゴリズムに対し、テストケース（フィクスチャ）の作成を成果物として明示している。具体的には以下の 3 ケースをカバーする:
   - 異なるレイヤーの衝突ルール（レイヤー優先順位で解決）
   - 同一レイヤー・異なる priority（priority で解決）
   - 同一レイヤー・同一 priority の MUST vs MUST（halt パス）

   他プランでは検証は受入基準として記述されるが、テストそのものを成果物とするのは GPT プランのみである。

**制約・注意点**: SKILL.md の内部矛盾が残存するため、`compound_conditions` の MUST 自己違反は解消されない。ランナー実装に依存するため、ランナーが存在しない環境では効果がない。また、エイリアシング規則がスコープ識別子にのみ限定されており、将来の識別子追加時に規則の拡張が必要になる。

## 4. 比較分析

### 4.1 P1（衝突解決の曖昧性）に対するアプローチ比較

5 プランすべてが P1 の解決策として同一の 3 段階カスケード（レイヤー → priority → conflict policy）を採用しており、解決のロジックに本質的な差異はない。

差異があるのは「どこにそのロジックを実装するか」である:

| プラン  | 実装先     | `interpretation.priority` の更新 | `MUST_vs_MUST` の更新 |
| ------- | ---------- | -------------------------------- | --------------------- |
| Claude  | SKILL.md   | 必須                             | 必須                  |
| Codex   | SKILL.md   | 言及（具体文なし）               | 言及（具体文なし）    |
| Copilot | SKILL.md   | 任意（optional）                 | 必須                  |
| Cursor  | SKILL.md   | 必須                             | 必須                  |
| GPT     | ランナー側 | 変更なし                         | 変更なし              |

Copilot プランのみが `interpretation.priority` の更新を任意とし、GPT プランのみが SKILL.md を一切変更しない。

### 4.2 P2（条件識別子の語彙）に対するアプローチ比較

| プラン  | 正規化方向                                     | 識別子の大文字小文字   | 代替案の検討       | 追加施策                                    |
| ------- | ---------------------------------------------- | ---------------------- | ------------------ | ------------------------------------------- |
| Claude  | conditions → definitions（ハイフン区切り）     | 全小文字               | なし               | `interpretation.condition_identifiers` 追加 |
| Codex   | 方向は明示せず（統一を要求）                   | 指定なし               | なし               | 検証フェーズの独立化                        |
| Copilot | conditions → definitions（ハイフン区切り）推奨 | 大文字保持（AI, YAML） | あり（2 案を比較） | BNF 文法を将来検討                          |
| Cursor  | 2 案を並記、推奨なし                           | 指定なし               | あり（判断は保留） | なし                                        |
| GPT     | ランナー側エイリアシング                       | 変更なし               | なし               | 予約トリガーリスト                          |

識別子の命名規則における Claude（全小文字: `creating-ai-directive-file`）と Copilot（大文字保持: `creating-AI-directive-file`）の差異は、実装時に選択が必要となる。既存の definitions キーとの一貫性を重視するなら全小文字が優位だが、可読性を重視するなら大文字保持にも合理性がある。

## 5. 総括

5 本のプランは同一の課題（P1/P2）に対して、それぞれ異なる設計哲学を反映している。

**Claude プラン**は「実装仕様書」として最も完成度が高く、影響範囲の完全な棚卸しと再発防止策の提案が他にない強みである。実装者が追加調査を必要とする場面が最も少ない。

**Codex プラン**は品質管理プロセスとしての形式的正しさに優れる。特にベースライン棚卸しの exit condition 化と検証整合性フェーズの独立化は、大規模な仕様変更における品質保証の観点で有用である。

**Copilot プラン**は意思決定支援の面で最も充実しており、代替案の検討プロセスの可視化は、複数のステークホルダーが関与する場面で合意形成を助ける。大文字保持の命名判断と、P1 修正の他の conflict policy への波及可能性という視点は、他プランにはない。

**Cursor プラン**は最小のオーバーヘッドで問題の本質を捉えており、迅速な実行が求められる場面に適している。ただし、リスク評価や詳細設計が含まれないため、複雑な仕様変更には補完が必要である。

**GPT プラン**は最も異質であり、「仕様の矛盾は残したまま、ランナー側で決定論的に処理する」という発想は、仕様の直接修正が困難な状況（例: 複数のステークホルダー間で仕様変更の合意が得られない場合）において現実的な選択肢となりうる。テスト駆動の成果物定義とエイリアシング機構の提案は、ランナー実装を見据えた設計として独自の価値がある。
